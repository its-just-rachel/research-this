# ADR-001: Repository and Documentation Boundary

**Date:** 2026-07-06
**Status:** Accepted
**Deciders:** Tech Range Platform Team
**Supersedes:** —
**Superseded by:** —

---

## Context

Tech Range is a signal-first technology intelligence platform. As the spec set has grown — ground-truth doctrine, architectural decisions, technical specs, design specs, and eventually implementation code — the question of what lives where has real consequences. Documentation that drifts into the wrong repo becomes stale faster than it can be corrected. Implementation code committed alongside doctrine creates ambiguity about what is prescriptive vs. what is emergent from the build.

The `research-this` repository currently holds two categories of content:

- `tech_range_ground_truth_docs_v0_2/` — canonical ground-truth doctrine: mission, operating model, ontology, classification model, platform principles
- `tech_range_build_specs_v0_1/` — build-planning artifacts: ADRs, technical specs, and design specs

A third category — implementation code — does not yet exist in this repo but will. A fourth category — operational content (actual intelligence records, advisory drafts, signal data) — must never enter this repo.

This ADR defines the boundary rules governing what belongs in this repository, what belongs elsewhere, and what must never be committed anywhere.

---

## Decision

### What belongs in this repository

**1. Ground-truth doctrine** (`tech_range_ground_truth_docs_v0_2/` and successors)

Canonical descriptions of what Tech Range is, how it thinks, and what it produces. These documents are the authoritative source of intent. They change slowly and deliberately.

- Mission and operating model
- Platform principles
- Ontology and entity vocabulary
- Strategic classification model (Neighborhoods, Domain Areas)
- Technology Profile model and lifecycle
- Solution Candidate and Advisory models
- Fidelity × Availability framework

Ground-truth docs are version-controlled with a version directory suffix (e.g., `v0_2`). A new version directory is created when a breaking change to the model is made; the prior version is retained for reference.

**2. Build-planning specs** (`tech_range_build_specs_v0_1/` and successors)

Architectural decisions, technical specifications, and design specifications that translate ground-truth doctrine into buildable architecture. These change more frequently than doctrine — they are refined as implementation decisions are made.

- `architectural-decisions/` — ADR-001 through current; governs design decisions and their rationale
- `technical-specs/` — TS-01 through current; defines data model, pipeline, state machines, API contracts, security, observability, deployment
- `design-specs/` — DS-01 through current; defines UI layout, interaction patterns, role-differentiated behavior

**3. Implementation code** (to be added in a `src/` or equivalent directory)

Application source code, database migrations, infrastructure-as-code, and CI/CD configuration. Code lives alongside the specs it implements so that a developer can read the spec and the implementation in the same context.

Migrations must be numbered sequentially and named to match the TS-02 schema they implement. No migration may introduce a schema object not described in TS-02; TS-02 is updated first, migration follows.

### What does NOT belong in this repository

**Operational intelligence content** — actual Signal records, Advisory drafts, Technology Profile data, source lists, analyst notes, and any content generated by running the platform. This content belongs in the platform's database and the Sensitive Content Store (per ADR-012). It must never be committed to any code repository.

**Sensitive partner, source, or program details** — non-public signal sources, restricted program names, client mission cases, contract details, or controlled technical information. Where a spec must reference such a thing, use a placeholder: `[Internal example]`, `[Sensitive partner]`, `[Restricted program]`, `[Client mission case]`, `[Non-public signal source]`, `[Internal decision rule]`.

**Vendor credentials, API keys, and secrets** — all secrets are managed outside the repository (see TS-11). Any committed secret, even historical, requires immediate rotation and must be purged from git history.

**Exploratory or draft work that has not reached decision status** — working notes, brainstorming documents, and pre-decision explorations do not belong in the main branch. Use branches or external documents until a decision is ready to commit as an ADR or spec.

**Duplicate copies of ground-truth doctrine** — the ground-truth docs are the single source of truth. Specs may quote short excerpts and must cite the source document; they must not restate the full doctrine inline. When doctrine changes, only the ground-truth doc is updated; specs reference it.

## Naming and versioning conventions

### Spec IDs

All specs carry a stable ID that never changes once assigned:

| Prefix | Series | Example |
|---|---|---|
| `ADR-` | Architectural decisions | `ADR-016` |
| `TS-` | Technical specs | `TS-02` |
| `DS-` | Design specs | `DS-13` |

IDs are assigned sequentially. Gaps in the sequence indicate deferred or intentionally skipped specs (e.g., DS-10 is deferred to a later phase). IDs are never reused.

### Spec status vocabulary

| Status | Meaning |
|---|---|
| `Draft` | Under active development; not yet authoritative |
| `Proposed` | Complete enough to review; decisions not yet final |
| `Accepted` | Decision made; this spec governs implementation |
| `Superseded` | Replaced by a newer ADR or spec version; retained for history |
| `Deferred` | Intentionally not addressed in current phase |

### Branch and commit discipline

- `main` is the protected branch. All content on `main` is considered authoritative.
- `master` is the working branch for this project. Changes are merged to `main` when a coherent set of specs is complete and reviewed.
- Commit messages must identify which specs changed and what category of change was made (draft, fix, consistency update, new).

## Consequences

**Positive:**
- A developer reading a spec can trust it reflects current intent, not historical exploration.
- Sensitive content is never at risk of accidental commit.
- The relationship between doctrine (what Tech Range is) and specs (how to build it) remains navigable as the codebase grows.
- A new team member can onboard by reading this repo in order: ground truth → ADRs → TS → DS → src.

**Negative / accepted costs:**
- Discipline is required to keep operational content out of the repo. No automated enforcement exists at the repository layer (only cultural convention and code review).
- Placeholder syntax (`[Internal example]`) reduces specificity in some specs. Where a real example would aid implementation, a separate internal reference document (not committed here) should be maintained.
